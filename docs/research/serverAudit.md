## KEY FINDINGS
- Sound foundation overall: thin routers -> whitelist-based repos with injected Db -> pure sr.ts scheduler, parameterized SQL throughout, immutable answer_attempts log with transactional SRS cache mirroring; the main pre-roadmap blockers are the lack of any migration mechanism (schema.sql is CREATE IF NOT EXISTS only), no validation layer, and no central error middleware.
- Latent seed-idempotency bug: UNIQUE(objective_id, question_text) at db/schema.sql:55 treats NULLs as distinct, so the seed.ts:110-117 'inserting unlinked' path duplicates rows on every re-seed once any question's objective_title fails to resolve (currently zero unlinked in nca-genl.json, so it is armed but not firing).
- Timezone inconsistency: sr.ts writes local-time attempted_date/next_review_date while all created_at/updated_at defaults and bumps use UTC datetime('now'), so the same row can carry timestamps a day apart — must be resolved before per-question analytics.
- Update paths skip create-time validation: updateObjective (objectives.repo.ts:59-77), updateQuestion, and updateTemplate accept empty or non-string title/question_text, and client-writable date columns (objectives.next_review_date, lab_runs.started_at/completed_at) accept arbitrary text that breaks lexical date comparisons.
- Mock-exam/MCQ readiness requires ALTER TABLE on recall_questions (question_format) and answer_attempts (source, session_id, selected_choice_id), plus new exam_sessions/exam_items/question_choices/domains tables; domain weights currently live only in apps/web/src/pages/Dashboard.tsx:13 and objectives.domain is freeform TEXT.
- LLM grading fits cleanly as a new attempt_gradings table, but answer_attempts.rating NOT NULL (db/schema.sql:77) blocks a grade-before-self-rate flow, and user_answer is optional so many attempts have nothing to grade.
- recordAttempt (reviews.repo.ts:43-91) unconditionally rewrites the SRS cache — if mock-exam answers reuse answer_attempts, an exam run would reschedule every card; a source discriminator and conditional mirror are needed.
- Error handling is uneven: GET /api/dashboard and /api/reviews/due have no try/catch (default HTML 500s), raw SQLite constraint messages leak as 400 bodies, status mapping is by substring match (routes/reviews.ts:34), and :id params are never validated (GET /api/reviews/history/abc returns 200 []).
- Test suite is genuinely good (in-memory DB repo tests covering CHECKs, FK cascades, cache mirroring, fixed clocks; sr.test.ts covers DST/leap boundaries) but has zero seed.ts tests (a double-seed test would catch the B1 bug), smoke-only HTTP coverage with no PUT/DELETE/reviews-due routes, and routes.test.ts:4 relies on fragile ESM import-hoisting luck with DB_PATH (db.ts:19 eagerly captures the real path).
- Dashboard blind spot: questions orphaned to objective_id NULL are counted in global due totals but invisible in every per-domain rollup (dashboard.repo.ts:23-31 vs 58-66), so domain dueCounts do not sum to the global figure.

# LLMStudy Backend Deep-Read Report

## 1. Architecture and code quality — is this a sound foundation?

**Verdict: yes, with reservations.** The backend is a small, disciplined three-layer design that will support the roadmap features, but three cross-cutting gaps (no migration story, no validation layer, no central error handling) should be fixed *before* building mock exams / LLM grading on top.

**What's good:**

- **Clean layering.** Thin Express routers (`apps/server/src/routes/*.ts`) → repos that take an explicit `Db` handle (`objectives.repo.ts`, `questions.repo.ts`, `reviews.repo.ts`, `lab-templates.repo.ts`, `lab-runs.repo.ts`, `dashboard.repo.ts`) → one pure-logic module (`sr.ts`). `createApp()` in `apps/server/src/app.ts:10` has zero DB/network side effects; all DB access goes through the injected handle, which is what makes the in-memory test suite possible.
- **SQL hygiene.** Every repo builds INSERT/UPDATE column lists exclusively from a hardcoded `WRITABLE` whitelist (e.g. `objectives.repo.ts:6-27`, `lab-runs.repo.ts:4-29`), and all values are bound parameters. No injection vector; unknown body keys are silently dropped. The only string-interpolated table name is a TypeScript literal-union (`seed.ts:169-177`), which is safe.
- **Good data-modeling instincts.** Immutable `answer_attempts` history + denormalized SRS cache on `recall_questions` (documented at `db/schema.sql:40-43, 61-71`), mirrored inside a single transaction (`reviews.repo.ts:57-89`). FK behaviors are deliberate: attempts CASCADE with their question, lab runs SET NULL so history survives template/objective deletion (`db/schema.sql:74, 129-132`). `UNIQUE` constraints exist specifically to make seeding idempotent (`db/schema.sql:31, 55, 107`).
- **Pure scheduler.** `sr.ts` is dependency-free, injectable-clock (`todayIso(now: Date = new Date())`), and DST-safe via UTC calendar arithmetic (`sr.ts:49-54`) — easy to swap for SM-2 later, exactly as the header comment intends.
- **Sensible ops story for an MVP.** WAL mode + FK pragma at open (`db.ts:31-37`), auto-seed only on truly empty DB (`index.ts:12-21`), idempotent `db/schema.sql` applied on every boot, `bin/reset.ts` deletes WAL/SHM sidecars too.

**Structural weaknesses:**

- **No migration mechanism.** `db/schema.sql` is 100% `CREATE ... IF NOT EXISTS` (acknowledged at `db/schema.sql:1-4`). That works until any roadmap feature needs to *alter an existing table* (e.g. adding `question_format` to `recall_questions` or `source` to `answer_attempts`) — which mock-exam/MCQ almost certainly does. There is no `PRAGMA user_version` scaffolding. This is the single biggest thing to add before roadmap work; otherwise every schema-touching feature forces `db:reset` and destroys user study history.
- **No validation layer.** Validation is an ad-hoc mix of `typeof` checks for required fields (`objectives.repo.ts:48`, `questions.repo.ts:55`) plus DB CHECK constraints for the rest. There's no zod/valibot at the route boundary. Consequences: raw SQLite error strings leak to clients as 400 bodies (e.g. `"CHECK constraint failed: confidence BETWEEN 1 AND 5"`), a JSON boolean in any numeric field yields better-sqlite3's cryptic bind error, and date-typed columns accept arbitrary text (see bug B4).
- **No central error handling.** Each router duplicates a private `messageOf()` (`routes/objectives.ts:48-50`, `routes/questions.ts:59-61`, `routes/lab-templates.ts:55-57`, `routes/lab-runs.ts:68-70`), and GET handlers have no try/catch at all (`routes/dashboard.ts:8-10`, `routes/reviews.ts:8-15`), so an unexpected throw returns Express's default HTML 500 page. Error-to-status mapping by substring match (`routes/reviews.ts:34`: `message.includes('not found') ? 404 : 400`) is fragile. An Express error middleware + typed domain errors would fix all of this in ~30 lines.
- **DB singleton is never closable/resettable** (`db.ts:39-45`) — fine now, but LLM-grading background work or a test that needs a fresh process-wide DB will fight it.

## 2. Concrete bugs, edge cases, and design flaws

Ordered roughly by severity.

**B1 — Latent seed-idempotency bug for unlinked questions.** `recall_questions` dedupes on `UNIQUE (objective_id, question_text)` (`db/schema.sql:55`), but SQLite treats NULLs as distinct in unique indexes. `seedQuestions` explicitly inserts with `objective_id = NULL` when a question's `objective_title` doesn't resolve (`seed.ts:110-117`, the "inserting unlinked" warning path). For any such row, `INSERT OR IGNORE` never ignores — **every re-run of `npm run seed` inserts another duplicate**. Currently harmless (I verified `db/seed/nca-genl.json`: all 100 questions reference one of the 25 objective titles, zero unlinked), but it's armed: one typo'd `objective_title` in a future seed edit, or an objective deleted directly in SQL (its questions go NULL via `ON DELETE SET NULL`), and re-seeding silently multiplies rows. Fix: dedupe by `question_text` when `objective_id IS NULL` (partial unique index `WHERE objective_id IS NULL`), or check-before-insert in `seed.ts`.

**B2 — Mixed local-time and UTC timestamps in the same tables.** `sr.ts:33-43` (`todayIso`, `localTimestamp`) produce **local** wall-clock values used for `answer_attempts.attempted_date`, `next_review_date`, and the question cache (`reviews.repo.ts:53-55`), while every `created_at`/`updated_at` default and bump uses SQLite `datetime('now')`, which is **UTC** (`db/schema.sql:29-30, 53-54, 78-80`; `reviews.repo.ts:78`; all repo UPDATE statements). So a single `answer_attempts` row written at 8pm in a UTC-7 timezone has `attempted_date = 2026-07-09 20:00:00` and `created_at = 2026-07-10 03:00:00` — different days. Within-column comparisons stay consistent (the due queue and `last7Days` both compare local-vs-local), so nothing user-visible breaks today, but any future analytics joining or sorting across these columns (per-question analytics is on the roadmap) will be off by up to a day. Pick one convention (local is the right one for a study app's "day" semantics) and apply it to `created_at`/`updated_at` too, or store UTC everywhere plus a timezone.

**B3 — Update paths skip create-time validation.** `createObjective` requires a non-empty string title (`objectives.repo.ts:48-50`) but `updateObjective` (`objectives.repo.ts:59-77`) happily sets `title: ""` or `title: 123` (SQLite TEXT affinity stores the number). Same asymmetry for `question_text` in `updateQuestion` (`questions.repo.ts:66-84`) and `title` in `updateTemplate` (`lab-templates.repo.ts:133-157`). `PUT /api/objectives/1 {"title": ""}` corrupts a blueprint row that the UNIQUE(cert_path,title) seed-idempotency then keys on.

**B4 — No format validation on client-writable date columns.** `objectives.last_reviewed_date` / `next_review_date` (writable via `objectives.repo.ts:13-14`) and `lab_runs.started_at` / `completed_at` (writable via `lab-runs.repo.ts:16-17`) accept any string — `"July 4th"`, `"tomorrow"`. All date logic is lexical string comparison (`sr.ts:66`, `dashboard.repo.ts:28, 62`), so one malformed value silently breaks due/overdue computations for that row. A one-line `^\d{4}-\d{2}-\d{2}` check in the repos (or zod at the boundary) closes this.

**B5 — Fragile test/env bootstrapping around `DB_PATH`.** `db.ts:19` eagerly computes `export const DB_PATH = resolveDbPath()` at module load, directly under a comment (line 13) claiming lazy resolution. `routes.test.ts:1-9` sets `process.env.DB_PATH = ':memory:'` "before importing db" — but ESM import hoisting means db.ts evaluates *first*; the test only works because `getDb()`/`openDb()` re-read the env lazily (`db.ts:31, 42-45`). If anyone ever makes db.ts eagerly open the singleton, or uses the exported `DB_PATH` constant in the server path, **the test suite will silently read/write the real on-disk `db/data/study.db`**. Either delete the `DB_PATH` export (only `bin/reset.ts:3` uses it) or make it a function.

**B6 — Unhandled-throw GET routes return HTML 500s.** `routes/dashboard.ts:8-10` and `routes/reviews.ts:8-15` have no try/catch and there is no error middleware in `app.ts`. Any repo throw (corrupt row, disk error) yields Express's default HTML stack-trace page instead of JSON — a problem the moment the roadmap SPA expects JSON error shapes everywhere.

**B7 — `:id` params are never validated.** `Number("abc")` is NaN; better-sqlite3 binds NaN as NULL so `WHERE id = NULL` matches nothing and you get a 404 — acceptable by accident. But `GET /api/reviews/history/abc` returns **200 `[]`** (`routes/reviews.ts:13-15`), indistinguishable from a real question with no history; `listHistory` (`reviews.repo.ts:30-34`) never checks the question exists. Also `GET /api/reviews/history/9999` for a nonexistent question → 200 `[]`.

**B8 — Query-param edge cases.** `numParam(' ')` → `Number(' ')` = `0` (`routes/lab-runs.ts:64-66`), silently filtering by `template_id = 0` (empty result) instead of 400. `objective_id=1.5` passes NaN checks in `routes/questions.ts:17-20` and filters by 1.5. `status=bogus` on lab-runs returns 200 `[]` rather than 400 against the known enum. All minor for a single-user app.

**B9 — No `DELETE /api/objectives/:id`.** Questions, templates, and runs all have DELETE; objectives don't (`routes/objectives.ts` has only GET/POST/PUT). If intentional (protecting blueprint + seed idempotency), it deserves a comment; note that deleting objectives out-of-band is exactly what arms bug B1.

**B10 — Dashboard blind spot: unlinked questions.** Per-domain `questionCount`/`dueCount` aggregate only questions with a non-NULL `objective_id` (`dashboard.repo.ts:23-31`), while the global totals count all questions (`dashboard.repo.ts:58-66`). A question orphaned by objective deletion is due-counted globally but invisible in every domain row — domain dueCounts won't sum to the global figure and no "Uncategorized" bucket catches them (that bucket only exists for objectives with NULL domain, `dashboard.repo.ts:115`).

**B11 — Flat SRS intervals (design note, documented).** A card rated 5 forever repeats every 14 days (`sr.ts:12-18`) — no ease growth. Fine per the spec comment, but with 100 seeded questions the steady-state daily queue will grow uncomfortably; the pure-function isolation makes an SM-2 swap cheap.

**B12 — Micro-inefficiencies (all fine at this scale, listed for completeness).** `listTemplates` loads the tag map for *all* templates even when filtered by objective (`lab-templates.repo.ts:97`); `dashboard.repo.ts:75` wraps `attempted_date` in `date()`, defeating any index, and there's no index on `answer_attempts(attempted_date)` anyway (only `(question_id, attempted_date)`, `db/schema.sql:83`); N+1-ish correlated subqueries in `OBJECTIVE_AGG` (`dashboard.repo.ts:23-31`) — irrelevant at 25 objectives.

## 3. Schema readiness for roadmap features

The schema's core bet — immutable `answer_attempts` as source of truth + cache on the question — is the *right* substrate for grading and analytics. What each feature needs:

**MCQ questions.** `recall_questions` is open-ended only. Needs: (a) `question_format TEXT NOT NULL DEFAULT 'open' CHECK (question_format IN ('open','mcq'))` on `recall_questions` — **this is an ALTER TABLE on an existing table, forcing the migration mechanism question first** (see §1); (b) a new `question_choices` table (`id, question_id FK ON DELETE CASCADE, choice_text NOT NULL, is_correct INTEGER NOT NULL DEFAULT 0, position INTEGER`) — new table, so idempotent-schema-compatible; (c) `answer_attempts` gains `selected_choice_id INTEGER REFERENCES question_choices(id)` (another ALTER). Note the actual NCA-GENL exam is MCQ, so this lands early in any mock-exam effort.

**Mock exam mode.** New tables (schema hooks named at `db/schema.sql:154-156` but nothing exists):
- `exam_sessions (id, cert_path, mode, question_count, started_at, completed_at, duration_seconds, score_pct)`
- `exam_items (id, session_id FK CASCADE, question_id FK SET NULL, position, user_answer, selected_choice_id, is_correct, time_spent_ms, flagged)`

Two design decisions the current code forces:
1. **Domain weights live only in the web client** (`apps/web/src/pages/Dashboard.tsx:11-13`, `DOMAIN_WEIGHTS`), and `objectives.domain` is freeform TEXT (`db/schema.sql:21`) with no domains table. Weighted exam assembly server-side needs a `domains (cert_path, name, weight)` table (also fixes the typo-risk of freeform domain strings and gives the dashboard one source of truth).
2. **Do exam answers create `answer_attempts` rows?** If yes, `recordAttempt` (`reviews.repo.ts:43-91`) will clobber the SRS cache — an exam blitz through 60 questions would reschedule every card. You'd need `source TEXT CHECK (source IN ('review','exam'))` + nullable `session_id` on `answer_attempts`, and make the cache-mirror conditional on `source='review'`. If no, keep exam answers solely in `exam_items` and accept split per-question history. Either way, decide before building — retrofitting is painful without migrations.

**LLM grading.** Cleanest fit of the roadmap. Add `attempt_gradings (id, attempt_id FK ON DELETE CASCADE, grader_model TEXT, rubric TEXT, score REAL, feedback TEXT, created_at)` — pure new table, no ALTERs. `expected_answer` (`db/schema.sql:48`) already serves as the rubric, and `user_answer` is captured per attempt. Two frictions: (a) `answer_attempts.rating` is `NOT NULL` (`db/schema.sql:77`), so a "grade first, then suggest a rating" flow can't insert the attempt before the user self-rates — either make rating nullable + a finalize step, or grade synchronously in `POST /api/reviews`; (b) `user_answer` is nullable and the UI treats typing an answer as optional (`routes/reviews.ts:29`), so many attempts will have nothing to grade — grading UX should require/encourage a typed answer.

**Per-question analytics.** Mostly ready: attempts are immutable, indexed by `(question_id, attempted_date)`. Missing: `duration_ms` on attempts (can't measure fluency), the `source` discriminator above, an index on `attempted_date` alone for time-window queries, and resolving B2 (mixed timezones) *before* charts are built on these columns.

**RAG study assistant.** Nothing exists and nothing conflicts. `documents / chunks / citations` are new tables; better-sqlite3's bundled SQLite includes FTS5 for keyword retrieval, and `db.loadExtension()` supports sqlite-vec for embeddings. The WAL comment at `db.ts:33` ("future Python sidecar co-read") suggests ingestion may happen out-of-process — the pragma choice already supports that.

**Bottom line for §3:** new-table features (grading, RAG, exam sessions) fit the current idempotent-schema approach; anything touching `recall_questions` or `answer_attempts` (MCQ, exam-source attempts, durations) requires adding a minimal `PRAGMA user_version` migration runner to `applySchema` (`db.ts:26-28`) first. That plus a `domains` table are the two concrete pre-roadmap schema tasks.

## 4. Test suite quality and gaps

**Quality: good for its size — real unit tests, not theater.** Every repo has an isolated in-memory-DB suite that exercises actual SQL: defaults and CHECK rejections (`objectives.repo.test.ts:55-57`, `lab-runs.repo.test.ts:47-53`), whitelist enforcement including an `id: 999` injection attempt (`objectives.repo.test.ts:39-49`), FK CASCADE/SET NULL behavior (`questions.repo.test.ts:75-80`, `reviews.repo.test.ts:78-83`, `lab-runs.repo.test.ts:77-85`, `lab-templates.repo.test.ts:61-75`), cache mirroring with a fixed clock (`reviews.repo.test.ts:23-35`), due-window boundaries (`reviews.repo.test.ts:62-67`), and dashboard aggregation including the empty-DB case (`dashboard.repo.test.ts:26-32`). `sr.test.ts` covers month/year/leap boundaries and time-component stripping (`sr.test.ts:25-30`). Injectable `Db` + injectable `now` is what makes all of this deterministic — a direct payoff of the architecture.

**Gaps, in priority order:**

1. **`seed.ts` has zero tests.** A "seed twice, counts unchanged" test is the single highest-value addition — it would immediately catch bug B1 the day an unlinked question appears, plus cover the unknown-`objective_title` warning path (`seed.ts:113-117`) and title→id linking. Seeding is also the only code path exercising `createTemplate` inside an outer transaction.
2. **`routes.test.ts` is smoke-only (6 tests).** No HTTP coverage of: any PUT or DELETE route, `GET /api/reviews/due` / `history`, any lab-runs route, query-param 400s (`objective_id=abc`), the 404-vs-400 mapping in `routes/reviews.ts:34`, or POST bodies with wrong-typed values (the cryptic bind-error path). The repo tests cover logic, but status-code/JSON-shape contracts are what the SPA depends on.
3. **No test pins the due-queue ordering** (`reviews.repo.ts:15`: overdue-scheduled first, new-material last) — an easy silent regression since it's a deliberate but uncommented product decision.
4. **Missing regression tests for known soft spots:** update-with-empty-title (B3) currently *passes* silently — a test asserting rejection would drive the fix; dashboard behavior for objective-less questions (B10); malformed date strings on writable date columns (B4).
5. **Fragile harness detail:** the env-ordering assumption in `routes.test.ts:1-9` (see B5) — converting `DB_PATH` to a function call removes the trap.
6. **No coverage of `index.ts` auto-seed guard or `bin/reset.ts`** — acceptable for CLI glue, but the auto-seed "only when objectives table is empty" rule (`index.ts:12-21`) is a behavior users depend on (deleted content staying deleted).

## Summary judgment

For a solo local-first MVP this is well above par: correct layering, parameterized SQL behind whitelists, an immutable event log where it matters, pure scheduling logic, and honest tests. It is a sound foundation for the roadmap **provided** four things land first: (1) a minimal `user_version` migration runner (blocks MCQ/exam-mode ALTERs), (2) a `domains` table so exam weights leave the web client, (3) a route-boundary validation layer + JSON error middleware, and (4) the timezone convention decision (B2) before any analytics feature reads timestamps. The two real latent bugs to fix now are the NULL-objective seed duplication (B1) and the unvalidated update paths (B3/B4).
